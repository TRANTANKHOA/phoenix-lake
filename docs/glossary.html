Glossary

Core concepts and technology choices with trade-offs.

ACID ↗
Atomicity, Consistency, Isolation, Durability.
Why here: Transactional consistency between data writes and catalog updates is the core invariant — either a snapshot is published or nothing is visible.
BEAM ↗
The Erlang/Elixir virtual machine. Excels at concurrency, messaging, and I/O; deliberately prevents one task from monopolising the machine.
Pros: Millions of concurrent connections, hot code upgrades, fault tolerance.
Cons: Not suited for CPU-bound analytical scans — which is why DuckDB is isolated from it.
Fit: BEAM handles the control plane's concurrency; DuckDB handles the data plane's CPU work.
DuckDB ↗
Embedded analytical SQL engine. One-writer, many-readers, single-process. Scans Parquet, joins, and aggregates in memory.
Pros: Fastest single-node analytical engine; no separate server to manage; native Parquet and DuckLake support.
Cons: Single-writer per database; can't shuffle across machines; concurrency bounded by RAM.
Fit: Perfect for workloads where one machine's RAM is enough — which is most small-to-mid analytics.
DuckLake ↗
Lakehouse table format storing metadata in a SQL database and data as Parquet on object storage. Provides snapshots, ACID with snapshot isolation, and schema evolution.
Pros: No separate catalog server; metadata latency drops from seconds to milliseconds; snapshots are cheap transactional rows.
Cons: Young ecosystem; third-party connectors (Spark, Trino, Athena) are work-in-progress; Iceberg has broader interop today.
Fit: When DuckDB is the primary engine and cross-engine interop isn't required.
Elixir / Phoenix ↗
Functional language and web framework built on the BEAM. Phoenix provides HTTP, WebSockets, LiveView for real-time UI, and the control-plane API.
Pros: Real-time UI with LiveView; Oban for reliable background jobs; excellent concurrency model; fault tolerance.
Cons: Smaller ecosystem than Node.js/Python; not suited for heavy computation.
Fit: Control plane owns orchestration, not computation — Elixir's strengths align perfectly.
Landing
First DuckLake database. Holds raw, validated data promoted from staging by the ingestion worker.
Fit: Source of truth for incoming data; all dbt models source from here via source().
Materialize
Process of refreshing pre-computed aggregations or reports from source tables on a schedule.
Fit: Reporting layer uses materialized views for dashboards — read-optimized, refreshed via Oban jobs.
MVCC ↗
Multi-Version Concurrency Control. DuckDB uses MVCC with optimistic concurrency control to allow concurrent reads without locking.
Fit: Readers never block writers; writers never block readers — matches the "one writer, unlimited readers" invariant.
Oban ↗
Background job framework for Elixir, backed by Postgres. Schedules work, controls concurrency, retries on failure, and persists jobs across restarts.
Pros: ACID-backed persistence; queue-based concurrency control; built-in retry and scheduling; Oban Pro adds workflows and rate limiting.
Cons: Requires Postgres; not a distributed task queue like Celery.
Fit: Serializes writers to one-per-table; queues split by cost profile (interactive, ingest, transform). Tells DuckDB what to do and when.
Parquet ↗
Columnar file format optimised for analytical queries. Immutable once written; the unit of data in DuckLake.
Pros: Columnar storage enables predicate pushdown; compression; widely supported.
Cons: Small files hurt scan throughput (compaction needed); immutable so updates produce new files.
Fit: Immutable files enable snapshot isolation — readers always see a consistent version.
Partition pruning ↗
DuckDB reads partition metadata to skip files that don't match query filters, avoiding full-table scans.
Fit: A 10 TB table may touch only a few GB per query — this is what keeps single-node scans fast.
Phoenix ↗
Elixir web framework. Provides HTTP, WebSockets, LiveView for real-time UI, and the control-plane API.
Pros: LiveView eliminates JavaScript for real-time features; channels for PubSub; strong security defaults.
Fit: Serves the API, LiveView UI, and orchestrates Oban jobs — the control plane's front door.
Postgres ↗
App metadata + three DuckLake catalogs (landing, refining, reporting). One database — transactions, snapshots, file lists, Oban jobs.
Pros: ACID transactions; mature ecosystem; DuckLake's first-class catalog database; Oban backend.
Cons: Single instance for both app metadata and catalogs — backup/recovery covers both.
Fit: Catalog update happens in the same transaction as data write — no window where files exist but catalog doesn't.
PubSub ↗
Phoenix publish-subscribe system. Broadcasts events (job completion, query results) to LiveView processes and API consumers.
Fit: LiveView updates in real-time when jobs complete or queries return — no polling needed.
Refining
Second DuckLake database. Holds transformed and enriched data written by dbt models or DuckDB transformation jobs.
Fit: Intermediate layer between raw landing and consumer-facing reporting — business logic lives here.
Reporting
Third DuckLake database. Holds materialized aggregations and pre-computed results for dashboards and downstream consumers.
Fit: Read-optimized layer for dashboards — refreshes via scheduled Oban jobs, readers always see latest snapshot.
S3 ↗
Object storage for Parquet data files. Enforced staging prefix for incoming files, plus three enforced DuckLake databases.
Pros: Effectively infinite storage; cost scales independently of compute; immutability by default.
Cons: Eventually consistent (but DuckLake handles this via catalog).
Fit: Storage decoupled from compute — many DuckDB processes can read the same Parquet concurrently.
Snapshot ↗
A commit to DuckLake that captures a consistent state of the database. Contains a changeset describing table creations, data inserts, deletes, and schema alterations.
Fit: Cheap to create (a few rows in metadata); enables time travel and consistent reads during writes.
Staging
A plain S3 prefix for incoming files. Not a DuckLake database. The ingestion worker reads from here and promotes to landing.
Fit: Decouples file delivery from ingestion — producers drop files, worker validates and promotes.
Write-then-publish ↗
The core invariant: write immutable Parquet to S3 first, then publish via a Postgres transaction. Readers always see a consistent snapshot.
Why: Files are immutable; publication is transactional — no partial writes, no inconsistent state. Concurrency reduces to one writer per table, unlimited readers.
