Glossary

Storage

Pluggable Parquet storage. S3, local disk, or Azure Blob — the adapter is swappable via config. Three enforced layers: staging, landing, refining, reporting.

1 Storage Adapter

Storage is pluggable. The DuckDB service talks to a storage interface — the implementation is swappable via environment variable.

Adapter pattern — storage is interchangeable

Storage backends

S3defaultAWS S3, MinIO, GCS (S3-compatible), Cloudflare R2. Production default.
Local diskdev / edgeNFS, EBS, local SSD. Fastest for single-node dev or edge deployments.
Azure BlobAzureAzure Blob Storage. For teams on Azure cloud.
Configenv varSTORAGE_BACKEND=s3|local|azure + STORAGE_PATH=...
DuckDB's httpfs extension handles S3 I/O natively. For local disk, DuckDB reads Parquet files directly — no extension needed. The adapter pattern is at the path resolution layer, not the I/O layer.

2 Three-Layer Template

Immutable Parquet across three enforced layers. Users choose what to put in it, not the structure.

Data flow through storage layers

Layer purposes

Staginglanding zonePlain prefix. Producers drop files here; ingestion worker reads and promotes. Cleaned up after promotion.
Landingraw dataRaw but validated data. Source of truth for incoming data. Readers query directly for raw analytics.
Refiningtransformeddbt models read from landing, apply joins/filters/business logic, write here.
ReportingaggregationsMaterialized aggregations for dashboards. Refreshed on schedule.

3 Layout Principles

Partition wisely

Partition by columns queries filter on — typically date and a small number of high-selectivity dimensions. Good partitioning lets DuckDB prune most files before reading any bytes.

Right-size files

Aim for 100MB–1GB files. Many tiny files destroy scan throughput; too many large files hurt parallelism. Compaction fixes this.

Immutable files

Updates and deletes produce new files and a new snapshot rather than mutating existing objects. This makes snapshots, time travel, and concurrent reads safe.

Decoupled from compute

Storage scales independently of compute. Many DuckDB processes can read the same Parquet concurrently with no coordination.

4 Snapshots & Cleanup

Lifecycle

  1. Each published snapshot references a specific set of files. Older snapshots keep referencing their older files, enabling time travel.
  2. A retention policy expires old snapshots. Once no live snapshot references a file, a cleanup job deletes it.
  3. Cleanup is always catalog-driven: a file is deleted only after the catalog confirms nothing points at it.
  4. Staging files are cleaned up by the ingestion worker after promotion or on a TTL-based sweep.
  5. Each database can have independent retention — landing may keep 90 days while reporting keeps 30.
Storage is cheap and scalable. Data volume grows independently of compute. No coordination overhead between readers — they each read immutable files.

5 Design Decisions

# Issue Decision Why
D1 S3-only vs pluggable storage Pluggable via adapter Supports S3, local disk, Azure Blob. Same Parquet files work everywhere.
D2 Catalog in SQL vs object-store manifests SQL (Postgres) ACID transactions, point-in-time recovery, familiar tooling.
D3 Three databases vs one with schemas Three DuckLake databases Independent retention, compaction, and access control per layer.
D4 File size target 100MB–1GB per file Small enough to prune, large enough to scan. Compaction enforces this.
