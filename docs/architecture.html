Glossary

System Architecture

Four components, two planes, one invariant.

The shape

System architecture — Phoenix Lake

🔮 Phoenix

API · LiveView · Oban · Auth. The control plane — long-lived workflows, concurrency, real-time UI, orchestration.

🦆 DuckDB service

Query execution · data ingestion · dbt transformation · Parquet management. The data plane — columnar scans, joins, aggregations.

🐘 Postgres

App metadata + three DuckLake catalogs (landing, refining, reporting). One database — transactions, snapshots, file lists, Oban jobs.

🪣 S3

A plain S3 prefix (staging) for incoming files, plus three enforced DuckLake databases (landing, refining, reporting). Immutable Parquet; no authoritative metadata.

Two planes

Control plane vs. data plane — Phoenix Lake

Control plane — Elixir

Owns everything concurrent, long-lived, or user-facing: HTTP and the UI, auth, the job scheduler, and the record of what work exists and its state.

Holds almost no data — it holds decisions about data, and streams results to users.

Data plane — DuckDB

Owns the CPU- and memory-heavy work: scanning Parquet, joining, aggregating, writing new Parquet.

Work runs in a long-running DuckDB service (warm, persistent connection) kept off the BEAM scheduler so one heavy query can't degrade the web tier.

Why isolate DuckDB from the BEAM? The BEAM excels at concurrency and I/O and deliberately refuses to let one task monopolise the machine — which is exactly what a large analytical scan does. Running DuckDB as a separate HTTP service keeps the web tier responsive and lets the two scale independently.

Infrastructure

Proposed components, grouped by role.

Compute
A Phoenix/Elixir cluster (web + Oban workers) and a DuckDB query service. The two scale on independent axes — web concurrency vs. query CPU/RAM.
Postgres
One managed instance: app metadata (owned by Phoenix) and three DuckLake catalogs — landing, refining, reporting — created automatically for every project.
S3
A plain S3 prefix (staging) for incoming files, plus three enforced DuckLake databases — landing (raw), refining (transformed), reporting (materialized). Users write models; the platform provides the structure.

Design validation

Checked against current DuckLake/DuckDB documentation (mid-2026). Core claims hold; three needed nuance.

Confirmed. Metadata in a SQL database (Postgres is first-class) with data as Parquet on object storage; catalog-in-SQL drops metadata latency from seconds to milliseconds and removes separate catalog servers; snapshots are cheap transactional rows (millions can coexist); full ACID with snapshot isolation including DDL; and DuckDB's one-writer/many-readers single-process model — which the design works with, not against.

Multi-writer is coming, not relied on beta

The Quack remote protocol turns DuckDB into a client/server DB with multiple writers, but is beta as of v1.5.2. The design serialises writers in the control plane instead.

Interop improving, still behind Iceberg WIP

DuckLake 1.0 lists Spark, Trino, DataFusion, Pandas clients and 0.3 added Iceberg interop — but most are work-in-progress and native Athena/Snowflake/BigQuery reads aren't there yet.

"No compaction" = metadata, not data clarified

Eliminating catalog servers and metadata compaction is real. Data-file compaction (merging small Parquet for scan speed) is still worthwhile and stays a DuckDB-service job.
