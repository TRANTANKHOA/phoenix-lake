Glossary

Data Flows

Three flows exercise the whole system. Each follows the same invariant.

Write-then-publish invariant: Write immutable Parquet to S3 first, then publish via a Postgres transaction. Readers always see a consistent snapshot — never a partial write.

1 Ingestion

Append-only. Data cleaning happens in dbt, not here.

Ingestion flow

File conventions

Path: s3://<bucket>/staging/<table>/

Name: <YYYYMMDD>_<HHMMSS>_<uuid4>.parquet

Non-conforming names tolerated. Timestamp extracted from Parquet metadata if needed.

Guarantees

Dedup: UUID + Oban uniqueness + catalog check

Schema: Field identifiers allow different schemas per file

Idempotent: Same file processed twice → same result

2 Query

Timeout-based routing — no upfront cost estimation.

Query routing

≤30s Sync timeout
202 Async response
∞ Concurrent reads

3 Transformation (dbt)

Git-synced models. User writes SQL; platform handles execution.

dbt transformation flow

dbt models source from landing, write to refining or reporting. Each model write publishes a new DuckLake snapshot — same write-then-publish invariant as ingestion.
