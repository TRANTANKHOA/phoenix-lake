Phoenix — Control Plane

Phoenix does no heavy data work; it coordinates. API, LiveView, Oban, Auth — the brain of the system.

API

Programmatic entry point for clients and integrations.

Request handling

  1. Accepts query requests, ingestion submissions, and job-control calls.
  2. Validates and authorizes every request before any work is scheduled.
  3. For fast, interactive queries, calls DuckDB service synchronously and streams rows back.
  4. For expensive or long-running work, enqueues an Oban job and returns a job handle.

Key rule: The API never blocks a web process on a large scan. Either the query is small and bounded, or it becomes a job.

Full API reference: See API Reference for endpoint schemas, request/response formats, and interactive Swagger UI.

Query routing

Timeout-based fallback — no upfront cost estimation needed.

Query routing decision

Routing logic

  1. Phoenix calls DuckDB service synchronously with a sync deadline (default: 5s).
  2. If query completes within the sync deadline, results stream back directly.
  3. If the sync deadline fires, Phoenix enqueues as Oban job, returns 202 with handle. A hard server-side kill (30s) terminates runaway queries.
  4. Client receives 202 with handle, then polls or subscribes to PubSub for results.
This avoids estimating query cost upfront. DuckDB service enforces per-query memory and row limits so runaway queries fail fast. Timeout is configurable per-project.

LiveView

Interactive UI — dashboards, query builders, job monitors, dataset browsers.

Real-time updates

Renders results pushed from the server. UI updates live as jobs progress or queries return — no polling needed.

PubSub integration

Subscribes to job and query events over PubSub. Running transformations and ingestions reflect live in the browser.

Server-side state

Keeps per-session UI state on the server. Suits exploratory analytics where each user drives a sequence of queries.

Minimal JavaScript

LiveView needs only a small JavaScript client (the WebSocket shim) — no separate JS frontend to build. Server-rendered, real-time, over WebSocket.

PubSub (multi-pod adapter)

LiveView and the API rely on PubSub for real-time job/query/ingest events. On one pod any adapter works; the choice is load-bearing only at phoenix.replicas > 1.

Adapter selection

Adapter
Phoenix.PubSub.PG2 — Phoenix default; zero infra, in-cluster broadcast; works on one pod or N.
Cluster wiring
libcluster (Kubernetes DNS / EPMD strategy), shared cookie. Required only at phoenix.replicas > 1 so nodes form a BEAM cluster.
Fallback (no clustering)
Phoenix.PubSub.Postgres (LISTEN/NOTIFY over the shared Postgres) when the BEAM can't be clustered — no Redis.
Extra infra (Redis)
Not added by default. Only if a Redis is already operated; the design avoids a second stateful service.
A job running on pod A must broadcast progress to a LiveView socket on pod B — PG2 + clustering is the multi-pod path; Postgres-backed PubSub is the no-extra-infra fallback.
Proposed, not yet built. Confirm the discovery strategy (libcluster Kubernetes.DNS vs Kubernetes.EPMD vs ECS/gossip) against the deployment target at scaffold time.

Oban

Job engine, backed by the same Postgres instance.

Queue configuration

interactive
Low-latency queries that timeout. Short-lived, high priority.
ingest
File ingestion and promotion. Serialized per table (one writer at a time).
transform
dbt model execution. Serialized per table. CPU and memory intensive.
maintenance
Compaction, snapshot expiration, file cleanup. Background, lower priority.
Queues separated by cost profile so a backlog of heavy jobs never starves quick ones. Each job invokes DuckDB service, records outcome, and publishes catalog snapshot.

Intelligence ownership — Phoenix decides WHEN

Compaction triggerOban cron/threshold"File count > 100" or "daily at 2am" → enqueue compact job
Partition suggestionIngestion workerAnalyze schema, suggest partition keys based on column types and cardinality
Retention expiryOban cron"Delete snapshots older than 90 days" → enqueue cleanup job
File cleanupOban cron"Delete staging files older than 24h" → enqueue cleanup job
Phoenix decides when to trigger maintenance. DuckDB decides how to execute it. Postgres stores the what (file metadata, snapshot versions).

Auth

Two authentication paths, one authorization model. All auth logic lives in the control plane.

Interactive Login

Browser-based, delegates to identity providers via Assent. Supports Google, WorkOS, Okta, and any OAuth2/OIDC provider.

  • User clicks "Sign in with Google" → redirect to IdP
  • IdP authenticates → callback with auth code
  • Phoenix exchanges code, fetches userinfo + groups
  • Find or create User by email, map groups → role
  • Set session cookie for subsequent requests
API Tokens

Programmatic access for CI, dbt runners, service accounts. Bearer tokens with bcrypt-hashed keys.

  • Client sends: Authorization: Bearer pk_live_...
  • Auth plug hashes token, looks up by prefix
  • Attaches user + grants to connection
  • Authorize plug checks dataset/table permissions

Roles

admin
Full access. Manage users, tokens, grants.
editor
Read + write data. Create/drop tables. Cannot manage users.
viewer
Read-only. Can run queries but not ingest.

Highest role wins when user belongs to multiple groups. Grants enforce dataset/table-level access regardless of auth method. See Auth Module for full design, flow diagrams, and code.

State in Postgres

App metadata owned by Phoenix

Users
Users, sessions, roles, and grants.
Datasets
Dataset and table registrations — the user-facing catalog of what exists.
History
Query history and results metadata.
Jobs
Oban job rows and their state.
App metadata is kept separate from — but in the same database as — the DuckLake catalog. See Postgres for details.
