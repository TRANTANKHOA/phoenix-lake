1 Deployment Topology

Stateless containers behind load balancer

Deployment model

Unit 1 container One DuckDB process per container (Axum + DuckDB)
Protocol PostgreSQL wire dbt connects via dbt-postgres adapter through load balancer. No custom adapter needed.
State Stateless No shared memory between containers. All state in Postgres + S3.
Reads Horizontal Add containers behind LB. Each handles independent queries. Scale on CPU > 70%.
Writes Serialized DuckDB service routes writes through single-writer queue per table. Oban prevents duplicate dbt runs.
Resources 8GB RAM, 4 CPU Per container. Tunable per workload.
Auto-scale HPA / ECS CPU > 70% → add container. CPU < 30% → remove container.

2 Storage Adapter

Storage is pluggable. DuckDB talks to a storage interface — the implementation is swappable.

Adapter pattern — storage is interchangeable

Storage backends

S3 default AWS S3, MinIO, GCS (S3-compatible), Cloudflare R2. Production default.
Local disk dev / edge NFS, EBS, local SSD. Fastest for single-node dev or edge deployments.
Azure Blob Azure Azure Blob Storage. For teams on Azure cloud.
Config env var STORAGE_BACKEND=s3|local|azure + STORAGE_PATH=...
DuckDB's httpfs extension handles S3 I/O natively. For local disk, DuckDB reads Parquet files directly — no extension needed. The adapter pattern is at the path resolution layer, not the I/O layer.

Append-only. Data cleaning happens in dbt, not during ingest.

Staging → validation → landing

Ingestion is idempotent: UUID in filename + Oban uniqueness + catalog-level dedup. Until the catalog transaction commits, readers see the previous snapshot.

7 Partitioning

DuckLake stores partition keys in catalog metadata, not file paths. Pruning via zone maps. DuckDB owns the execution intelligence.

Intelligence ownership — DuckDB decides HOW

Partition applyDuckDBReads partition keys from catalog, applies layout when writing Parquet
Partition pruneDuckDBReads zone maps (min/max stats), skips files that don't match query filters
Schema analysisDuckDBAnalyzes column types and cardinality to suggest partition keys
Transforms
identitycolLow-cardinality: region, status
bucketbucket(N, col)Hash distribution for skew
yearyear(ts)Annual
monthmonth(ts)Monthly (default for facts)
dayday(ts)Daily (100M+ rows/month)
hourhour(ts)Hourly (1B+ rows/day)
Grain guide
< 1M rows/monone or monthlySmall dimensions
1M–100M rows/momonthlyStandard facts
100M–1B rows/modailyHigh-volume events
1B+ rows/mohourly or bucketStreaming, IoT

Rules of thumb

  1. Partition on columns queries filter on, not columns they select.
  2. Target 100MB–1GB per partition — small enough to prune, large enough to scan.
  3. Avoid high-cardinality keys without bucketing — creates millions of tiny files.
  4. Dimension tables under 1GB rarely need partitioning.
  5. If a partition has <1000 rows, it's too granular — merge or remove.
  6. Composite partitions (tenant + time) work when queries always filter both.

8 Compaction

Merge many small Parquet files into fewer, larger files. Same data, fewer I/Os. DuckDB owns the execution intelligence.

Intelligence ownership — DuckDB decides HOW

File selectionDuckDBReads file metadata from catalog, identifies adjacent files to merge
Merge executionDuckDBReads Parquet, writes merged files, updates catalog in single transaction
Size optimizationDuckDBTargets 100MB–1GB per file based on file statistics

11 Threading Model

Concurrency spans two containers: Oban in Phoenix, query execution in DuckDB service.

Two-container threading

Phoenix container Oban limit: 2 How many dbt run workers execute concurrently. Workers are stateless — they submit SQL to DuckDB service.
dbt (in worker) threads: 2 How many models each worker submits to DuckDB service in parallel.
DuckDB service threads: 4 Morsel-Run Parallelism — each query splits across CPU cores inside DuckDB container.

Concurrency across containers

Recommended config

Container Setting Value Why
Phoenix Oban transform limit 2 2 concurrent dbt runs. Workers are stateless.
Worker dbt threads 2 2 models per run submitted to DuckDB service in parallel.
DuckDB DuckDB threads 4 Full core usage per query. Morsel parallelism handles I/O efficiently.

Workers scale independently from DuckDB containers. Add Phoenix containers for more concurrent dbt runs. Add DuckDB containers for more query throughput.

Don't fight thread counts. DuckDB's morsel parallelism is efficient at I/O-bound work (S3 reads). Oversubscription is fine when queries spend most time waiting on network, not CPU.

13 Security Boundary

Trust boundary: DuckDB trusts Phoenix completely

Security invariant: DuckDB has no authentication, no authorization, no user concept. It executes whatever SQL Phoenix sends. Phoenix is the single enforcement point — one boundary, checked once.

14 Design Decisions

# Issue Decision Why
D1 Rust vs Python for DuckDB service Rust (Axum + duckdb-rs) No GIL, lower memory, smaller image. Long-running compute engine favors performance over dev speed.
D7 Three-layer threading: Oban × dbt × DuckDB Oban: 2, dbt threads: 2, DuckDB threads: 4 Oversubscription is fine — queries are I/O-bound (S3). DuckDB morsel parallelism handles CPU contention efficiently.
D8 Intelligence split: Phoenix (when) vs DuckDB (how) Phoenix schedules, DuckDB executes Phoenix decides when to compact/partition (cron, thresholds). DuckDB decides how (file selection, merge strategy, size optimization).
D2 Write serialization: two-level model Oban (per-project, replace args) + DuckDB service (per-table) Oban replaces waiting job args on new push (latest wins). DuckDB service parses SQL, routes writes through single-writer queue per table.
D3 Read connection: per-request vs persistent Persistent read-only connection Catalog metadata cached in process memory. No re-initialization per query.
D4 Write connection: shared vs temporary Temporary per write operation No cross-operation state leakage. Clean separation between reads and writes.
D5 Partition keys in file paths vs catalog Catalog metadata only Partition layout can evolve without rewriting files. DuckLake handles mapping.
D6 Scaling reads vs writes Reads: horizontal (HPA). Writes: vertical. Reads are stateless containers. Writes bottleneck at Oban serialization, not DuckDB capacity.
