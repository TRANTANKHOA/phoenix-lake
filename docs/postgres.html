Postgres & DuckLake Catalog

One Postgres instance serves two roles: app metadata and the DuckLake catalog. Keeping them together is a deliberate simplification.

1Postgres instance
2Logical roles
3DuckLake databases
0Separate catalog services

1 Two Roles, One Database

One Postgres, two logical concerns

App metadata

Read and written by Phoenix on every request and job:

  • Users, sessions, roles, grants
  • Dataset/table registrations
  • Query history and results metadata
  • Oban job rows
DuckLake catalog

Lakehouse metadata stored in SQL, not object-store manifests:

  • Table definitions and schema evolution
  • Snapshot versions
  • Parquet file lists and statistics
  • Partition information for pruning

No overlap with DuckLake catalog. DuckLake manages its own internal tables (snapshots, data_files, etc.) — Phoenix never reads or writes them directly. Clean separation.

3 Three DuckLake Databases

Created automatically by the platform for every project.

Database creation

CREATE DATABASE landing   (TYPE ducklake, DATA_PATH 's3://<bucket>/landing/');
CREATE DATABASE refining  (TYPE ducklake, DATA_PATH 's3://<bucket>/refining/');
CREATE DATABASE reporting (TYPE ducklake, DATA_PATH 's3://<bucket>/reporting/');
Landing

Raw, validated data. Written only by the ingestion worker. Phoenix reads from it but treats it as owned by the data plane.

Refining

Transformed and enriched data. Written by dbt models and materialization jobs. Phoenix reads from it for query results and UI display.

Reporting

Materialized aggregations for dashboards. Scheduled Oban jobs or dbt external materializations refresh these on a cadence.

Isolation

Each database has independent snapshot retention, compaction policies, and access control. A bad dbt run in one database cannot corrupt another.

3 Snapshot Lifecycle

How data moves from staging to committed snapshot.

Write-then-publish invariant

sequenceDiagram participant Oban as Oban Worker (Phoenix) participant LB as Load Balancer participant DuckDB as DuckDB Service participant Storage as S3 / Local participant Postgres Oban->>LB: 1. submit SQL (COPY INTO, etc.) LB->>DuckDB: 2. route to container DuckDB->>Storage: 3. write Parquet files Storage-->>DuckDB: files persisted DuckDB->>Postgres: 4. BEGIN transaction DuckDB->>Postgres: 5. insert file refs into catalog DuckDB->>Postgres: 6. create new snapshot (version N+1) DuckDB->>Postgres: 7. update registry (row_count, size_bytes) Postgres-->>DuckDB: COMMIT DuckDB-->>LB: 8. success LB-->>Oban: 9. job complete

Write-then-publish. Parquet files are written to S3 first, then the catalog is updated in a single Postgres transaction. Readers see either the old snapshot or the new one — never a partial state.

4 Why One Postgres

No separate catalog service to deploy, secure, back up, and keep consistent. Backups, point-in-time recovery, and access control are solved once, for all roles, by Postgres.

Advantages

Transaction consistencyACIDCatalog update is one Postgres transaction — never orphaned files or phantom snapshots.
No extra service−1 componentNo Glue, no standalone metastore to deploy and maintain.
Backuppg_dump / PITROne backup strategy covers both app metadata and catalog.
Access controlPostgres rolesSame RBAC for app tables and catalog tables.

5 Snapshots & Cleanup

Lifecycle

  1. Each published snapshot references a specific set of files. Older snapshots keep referencing their older files, enabling time travel.
  2. A retention policy expires old snapshots. Once no live snapshot references a file, a cleanup job deletes it from storage.
  3. Cleanup is always catalog-driven: a file is deleted only after the catalog confirms nothing points at it.
  4. Staging files are cleaned up by the ingestion worker after promotion or on a TTL-based sweep.
  5. Each database can have independent retention — landing may keep 90 days while reporting keeps 30.

6 Maturity Caveat

DuckLake's catalog format is young. Third-party connectors (Spark, Trino, DataFusion) are emerging but most are work-in-progress. Native reads from Athena, Snowflake, and BigQuery aren't there yet. For broad multi-engine interop, Iceberg remains more mature.

8 Registry Sync

How DuckLake catalog and Phoenix registry stay consistent.

Write-through sync — same Postgres transaction

sequenceDiagram participant DuckDB as DuckDB Service participant Postgres DuckDB->>Postgres: BEGIN transaction DuckDB->>Postgres: UPDATE ducklake_catalog (internal) DuckDB->>Postgres: INSERT/UPDATE phoenix_registry (app tables) Postgres-->>DuckDB: COMMIT Note over DuckDB,Postgres: Atomic — both succeed or both fail

Sync triggers

CREATE TABLEDDLDuckDB service inserts into datasets + tables registry after DuckLake catalog update.
INSERT / COPYDMLOptionally update tables.row_count and tables.size_bytes after snapshot commit.
DROP TABLEDDLSoft-delete from registry (set deleted_at). DuckLake catalog handles file cleanup.
ALTER TABLEDDLUpdate tables.schema_json with new column definitions.
Same Postgres connection, same transaction. No polling, no events, no eventual consistency. The DuckDB service already connects to Postgres for DuckLake — it writes to both catalog and registry in one atomic commit.

9 Query Cost Capture

DuckDB exposes query metrics after execution. Capture them in query_history.

Metrics available from DuckDB

duration_mswall clockEnd - start timestamp. Primary cost metric.
rows_scannedI/OTotal rows read from Parquet files (before filtering).
rows_returnedoutputRows in final result set.
bytes_scannedI/OBytes read from storage. Direct proxy for S3 cost.
cpu_time_uscomputeCPU microseconds. Useful for comparing query efficiency.
memory_peak_bytesmemoryPeak memory during execution. Detects OOM risk.

How to capture

Option APost-exec querySELECT * FROM duckdb_query_metrics() after each query. DuckDB returns metrics for the last executed query.
Option BProfile callbackUse DuckDB's PRAGMA profiling_mode='detailed' + PRAGMA profiling_output='...' to capture structured output.
Option CAxum middlewareWrap query execution with timing + stats collection in Rust. Most control, no extra queries.
Recommendation: Option C (Axum middleware). The DuckDB service already wraps query execution — add timing + stats capture there. No extra Postgres round-trips. Store in query_history on the same connection.

10 Design Decisions

# Issue Decision Why
D1 One Postgres vs separate catalog service One Postgres for both Transaction consistency between data and catalog. Fewer components to deploy.
D2 DuckLake vs Iceberg catalog DuckLake Native DuckDB integration, simpler setup. Tradeoff: less ecosystem maturity.
D3 Three databases vs one with schemas Three DuckLake databases Independent retention, compaction, and access control per layer.
D4 Catalog in SQL vs object-store manifests SQL (Postgres) ACID transactions, point-in-time recovery, familiar tooling.
D5 Registry sync: polling vs write-through Write-through, same transaction DuckDB service writes to both DuckLake catalog and Phoenix registry in one Postgres transaction. No polling, no events, no eventual consistency.
D6 Query cost: extra query vs middleware Axum middleware in DuckDB service Capture timing + DuckDB stats in the Rust execution wrapper. No extra Postgres round-trips. Store in query_history.
