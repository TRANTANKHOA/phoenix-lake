Phoenix Lake

Greenfield "minimum viable lakehouse" — Elixir/Phoenix as control plane, DuckDB as engine, one Postgres for app metadata + three DuckLake catalogs (landing, refining, reporting), Parquet on pluggable storage (S3, local disk, Azure Blob).

Design proposal, not a deployed service. It does not replace the existing Iceberg + Athena + Glue platform — it targets workloads where the largest query intermediate fits in one machine's RAM and concurrency is bounded.

The shape

Four components, each aligned to its strength.

🔮 Phoenix

API · LiveView · Oban · Auth. The control plane — long-lived workflows, concurrency, real-time UI, orchestration.
🦆 DuckDB service

Query execution · data ingestion · dbt transformation · Parquet management. The data plane — columnar scans, joins, aggregations.
🐘 Postgres

App metadata + three DuckLake catalogs (landing, refining, reporting). One database — transactions, snapshots, file lists, Oban jobs.
🪣 S3

A plain S3 prefix (staging) for incoming files, plus three enforced DuckLake catalogs (landing, refining, reporting). Immutable Parquet; no authoritative metadata.
