Glossary

dbt Integration

Users bring their own dbt models via Git-synced repos. The platform provides the execution environment; the user provides the SQL.

Git-synced dbt workflow

User provides

GitHub/GitLab repo with a standard dbt project: dbt_project.yml, ducklake.yml, and model SQL files.

Models source from landing, write to refining or reporting via external materialization.

Platform provides

DuckDB execution, S3 credentials, DuckLake connections, profiles.yml generation, resource isolation, scheduling (editable per-project cron), test-result surfacing, and retry logic.

Each run executes in an Oban worker that submits SQL to the long-running DuckDB service over HTTP; the service enforces per-query memory and time caps.

Git-synced dbt flow

  1. User connects their GitHub/GitLab repo via the Phoenix UI and selects a branch.
  2. Platform reads ducklake.yml from the repo root to provision catalogs and set retention policies.
  3. Platform validates the dbt project against the enforced template: all sources must target landing, all outputs must target refining or reporting.
  4. A git push, webhook, cron schedule, or manual trigger enqueues an Oban job on the transform queue.
  5. The worker clones the repo, generates a profiles.yml pointing a thin custom adapter at the DuckDB service over HTTP, and runs dbt run.
  6. dbt reads from landing via source(), transforms, and writes to refining or reporting — each model write publishes a new DuckLake snapshot.
  7. Results are broadcast via PubSub and surfaced in the LiveView UI. Temporary files are cleaned up.

Template YAML — ducklake.yml

Every dbt project must include this file at the repo root. The platform reads it to provision catalogs and validate models.

layers:
  landing:
    description: "Raw validated data from ingestion"
    retention_days: 90
    partition_columns:
      - name: year
        expression: "year(ordered_at)"
      - name: month
        expression: "month(ordered_at)"
      - name: day
        expression: "day(ordered_at)"
  refining:
    description: "Transformed and enriched data"
    retention_days: 90
    partition_columns:
      - name: year
        expression: "year(ordered_at)"
      - name: month
        expression: "month(ordered_at)"
  reporting:
    description: "Materialized aggregations for consumption"
    retention_days: 30
    partition_columns: []

Template validation

Every project is validated on registration and on each sync. It must pass all of these before it runs:

  1. ducklake.yml exists at the repo root and is valid YAML.
  2. All three layers defined — landing, refining, reporting.
  3. Required fields present — each layer has a description.
  4. dbt_project.yml exists and is parseable.
  5. All source() references target landing — models may only read from the landing catalog.
  6. All ref() outputs target refining or reporting — writes to landing are not allowed.
  7. Reporting models use external materialization — Parquet to s3://<bucket>/reporting/....
  8. No hardcoded S3 paths or credentials — the platform injects secrets and data paths.

If validation fails, the project is rejected with a clear error and is not run until it passes.

Platform execution — profiles.yml

The worker generates a profiles.yml that points a thin custom adapter at the DuckDB service over HTTP. The adapter is a small fork of dbt-duckdb that overrides only the connection layer — it inherits DuckDB dialect, DuckLake ATTACH, and external materialization, and swaps the transport to submit compiled SQL over HTTP instead of opening a local DuckDB connection.

default:
  outputs:
    dev:
      type: duckdb_service        # custom adapter (dbt-duckdb fork)
      host: duckdb-service         # DuckDB service HTTP endpoint
      port: 8080
      # S3 credentials and data paths are injected by the service; the service
      # has all three DuckLake catalogs (landing, refining, reporting) ATTACHed.

Because the service has all three catalogs ATTACHed, source('landing', ...) and ref() resolve to catalog-qualified names the service routes correctly — there is no per-run dbname switch. DuckDB is never embedded in the BEAM or the Oban worker.

dbt project convention — catalog mapping

Users map their dbt package schemas onto the platform catalogs so sources and refs resolve correctly:

# dbt_project.yml
name: 'my_project'
version: '1.0.0'

models:
  my_project:
    staging:
      +schema: landing
    marts:
      +schema: refining

Incremental models

dbt Core v2 (Fusion) supports incremental strategies (merge, append, delete+insert, microbatch) on DuckLake tables. The platform supports these out of the box — no special configuration needed. microbatch is particularly effective for partitioned DuckLake tables.

Full design: See 09 — dbt integration for project conventions, isolation, failure handling, and incremental model support.
Catalog creation. The three catalogs (landing, refining, reporting) are created automatically by the platform at project setup. Users don't need to set up catalogs — they just write dbt models that target them.
