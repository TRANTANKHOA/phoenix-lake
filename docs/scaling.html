Glossary

Scaling

Reads auto-scale, writes are serialized. The honest decision rule for when this model fits.

Scaling model — reads auto-scale, writes are serialized

Reads — auto-scale

DuckDB containers are stateless for reads. Add/remove based on load:

  • CPU > 70% → add container
  • CPU < 30% → remove container
  • P95 latency > SLA → add container
  • Queue depth > threshold → add container

Kubernetes HPA or ECS target tracking. Default 1; scales 1→10 (set min 2 for high availability).

Writes — manual scale

Writes serialized by Oban (one writer per table). Adding containers doesn't help. Scale by:

  • Increasing Oban queue limits
  • Optimizing write operations
  • Splitting large tables

Bottleneck is Oban serialization, not DuckDB capacity.

Scaling boundaries

This design bets one machine is enough for the data plane. That bet is right more often than people assume — but it has real edges.

The single real limit: query intermediate size. Scans stream and don't accumulate, so scan size is rarely the problem (pruning means a 10 TB table may touch a few GB). The wall appears when a query must materialise state larger than one machine's RAM — a wide high-cardinality GROUP BY, a huge hash-join build side, a global sort. DuckDB spills to local disk; it can't shuffle across machines. The predictor is not table size, it's the largest intermediate a query must hold.
The limit you hit first: concurrency. DuckDB is single-writer per database and not a multi-tenant query server. This design handles bounded concurrency — tens of dashboards, a team of analysts — via a long-running DuckDB service with per-query limits and serialised writers. It does not scale to hundreds of simultaneous interactive tenants; that's a job for Snowflake / BigQuery / Trino.

The honest decision rule

✓

Use this design when

  • Largest intermediate fits in one machine's RAM
  • Concurrency is bounded (tens of dashboards, team of analysts)
  • No mature cross-engine interop required

Then it isn't just viable — it's simpler than the alternatives.

✗

Keep a distributed engine when

  • Cross-machine shuffle is the norm
  • Hundreds of concurrent tenants
  • Hardened multi-engine interop (Athena/Snowflake/BigQuery)

For the existing platform-data-lake — Iceberg + Athena + Glue — this branch holds.

The interesting wedge for this design is the small workloads currently paying the distributed-engine tax — single-partition transforms, modest refine models, embedded reporting — where one box is genuinely plenty.
